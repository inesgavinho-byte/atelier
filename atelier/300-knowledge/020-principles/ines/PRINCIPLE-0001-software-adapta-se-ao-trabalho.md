# PRINCIPLE-0001 — O software deve adaptar-se ao trabalho

## Enunciado

O software deve adaptar-se à forma como o trabalho acontece.

O trabalho nunca deverá ser alterado apenas para satisfazer limitações do software.

---

# Significado

Ao longo das últimas décadas, a maioria das plataformas de produtividade obrigou as pessoas a reorganizar a sua forma de trabalhar.

Criaram novos conceitos, novas estruturas e novos fluxos porque eram convenientes para o software, não porque refletiam a realidade.

O resultado foi simples:

As pessoas passaram a trabalhar para a ferramenta.

Não com a ferramenta.

O ATELIER rejeita esta abordagem.

O seu papel é observar como o trabalho realmente acontece e adaptar-se continuamente a essa realidade.

Se existir conflito entre o modelo mental do utilizador e a estrutura do software, é a estrutura do software que deverá evoluir.

---

# Fundamentação

O trabalho intelectual não é linear.

Uma ideia pode surgir após dias de observação inconsciente.

Uma decisão pode depender de uma conversa ocorrida meses antes.

Uma investigação pode transformar um projeto aparentemente sem relação.

O software não deve impor uma sequência artificial.

Deve respeitar a natureza do pensamento humano.

---

# Quando aplicar

Este Princípio deve orientar decisões de:

* arquitetura do produto;
* desenho da interface;
* experiência de utilização;
* organização da informação;
* automação;
* fluxos de trabalho;
* integração entre sistemas.

Sempre que uma funcionalidade obrigar o utilizador a alterar a sua forma natural de trabalhar, a solução deverá ser reavaliada.

---

# Quando não aplicar

Este Princípio não significa ausência de estrutura.

Uma organização consistente continua a ser necessária.

Também não significa reproduzir hábitos ineficientes apenas porque são familiares.

O objetivo é respeitar a forma como o trabalho realmente acontece, eliminando fricção desnecessária.

---

# Exemplos

## Exemplo 1

Uma ideia surge enquanto o utilizador está a trabalhar num Paper.

O sistema deverá permitir capturá-la imediatamente.

Não deverá obrigar o utilizador a escolher previamente um projeto, uma categoria ou uma pasta.

A organização acontece depois.

---

## Exemplo 2

Uma conversa iniciada com um modelo de IA pertence ao contexto de um projeto.

Não pertence ao fornecedor da IA.

Se amanhã o modelo mudar, a conversa continua integrada no mesmo contexto.

---

## Exemplo 3

Uma decisão pode relacionar-se simultaneamente com PAPERS, GAVINHO e DECIMA.

O sistema deverá permitir múltiplas relações.

Nunca deverá obrigar a escolher apenas uma localização.

---

# Consequências

Este Princípio conduz naturalmente a:

* captura universal;
* memória baseada em relações;
* contexto persistente;
* integração entre ferramentas;
* redução de carga cognitiva;
* menor dependência de fornecedores tecnológicos.

---

# Relações

## Reforça

* Primeiro executar. Melhorar depois da utilização real.

Todas as restantes decisões de produto deverão ser compatíveis com ele.

Sempre que existir conflito entre convenções de software e a realidade do trabalho, prevalece este Princípio.
